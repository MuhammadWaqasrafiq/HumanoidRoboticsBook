Section 1: URDF Structure​

Question 1: URDF Tree Structure​

Which statement about URDF structure is TRUE?

A) A URDF can have multiple root links B) Links can form cycles (loops) in the structure C) Every URDF must have exactly one root link with no parent D) Joints are optional if links don't need to move

Show Answer

Correct Answer: C) Every URDF must have exactly one root link with no parent

Explanation: URDF represents robots as a tree structure, not a graph. This means:

  • Exactly one root link (typically base_link)
  • No cycles allowed (each link has only one parent)
  • All other links connect via joints from the root

Why? Tree structures ensure unique transform paths from any link to any other link, which is essential for computing forward kinematics and TF transforms.

Invalid example (multiple roots):

<link name="root1"/>  <!-- No parent -->
<link name="root2"/>  <!-- No parent -->
<!-- ERROR: Two root links! -->

Valid example:

<link name="base_link"/>  <!-- Root -->
<joint name="j1" type="fixed">
  <parent link="base_link"/>
  <child link="child1"/>
</joint>
<link name="child1"/>

Reference: See urdf-basics.mdx:60-75 for URDF tree structure.

Question 2: Visual vs Collision Geometry​

You're modeling a robot hand with a detailed mesh (50,000 triangles). What's the best practice?

A) Use the detailed mesh for both visual and collision B) Use the detailed mesh for visual, simple primitives for collision C) Use simple primitives for both visual and collision D) Only define visual geometry (collision is optional)

Show Answer

Correct Answer: B) Use the detailed mesh for visual, simple primitives for collision

Explanation:

Visual geometry: What you SEE in rviz2/simulators

  • Can be very detailed (high-poly meshes)
  • Only affects rendering performance
  • Makes robot look realistic

Collision geometry: What physics engine USES

  • Should be simple (boxes, cylinders, spheres)
  • Affects simulation speed (checked thousands of times per second)
  • Complex meshes cause 10-100x slowdown

Best practice example:

<link name="robot_hand">
  <!-- Visual: detailed for appearance -->
  <visual>
    <geometry>
      <mesh filename="package://my_robot/meshes/hand_detailed.dae"/>
    </geometry>
  </visual>

  <!-- Collision: simplified for performance -->
  <collision>
    <geometry>
      <box size="0.15 0.08 0.05"/>  <!-- approximate hand as box -->
    </geometry>
  </collision>
</link>

Why option D is wrong: Collision geometry is required for simulation. Without it, the robot won't interact with the environment physically.

Reference: See urdf-basics.mdx:221-259 for visual vs collision geometry.

Section 2: Joints & Transforms​

Question 3: Joint Types​

Which joint type should you use for a robot's elbow?

A) continuous - unlimited rotation B) revolute - rotation with limits C) prismatic - sliding motion D) fixed - no movement

Show Answer

Correct Answer: B) revolute - rotation with limits

Explanation:

Joint types and use cases:

TypeDescriptionExample
revoluteHinge with limitsElbow (0 to 150), knee, finger joints
continuousUnlimited rotationWheels, propellers
prismaticLinear slidingTelescoping arm, elevator
fixedNo movementCamera mount, sensors

Elbow example:

<joint name="elbow" type="revolute">
  <parent link="upper_arm"/>
  <child link="forearm"/>
  <origin xyz="0 0 -0.3" rpy="0 0 0"/>
  <axis xyz="0 1 0"/>  <!-- pitch rotation -->
  <limit lower="0" upper="2.618" effort="30" velocity="2.0"/>
  <!-- 0 to 150 degrees, can't hyperextend -->
</joint>

Why not continuous? Human elbows can't rotate 360 - they have physical limits (0 to ~150).

Why not prismatic? Elbows rotate (hinge), they don't slide.

Reference: See urdf-basics.mdx:367-382 for joint types.

Question 4: Understanding <origin xyz rpy>​

Given this joint definition:

<joint name="shoulder" type="revolute">
  <parent link="torso"/>
  <child link="upper_arm"/>
  <origin xyz="0 0.2 0.5" rpy="0 0 0"/>
  <axis xyz="0 1 0"/>
</joint>

What does xyz="0 0.2 0.5" mean?

A) The upper_arm is 0.2m forward and 0.5m up from torso origin B) The upper_arm is 0.2m left and 0.5m up from torso origin C) The upper_arm is at world coordinates (0, 0.2, 0.5) D) The upper_arm is 0.2m right and 0.5m down from torso origin

Show Answer

Correct Answer: B) The upper_arm is 0.2m left and 0.5m up from torso origin

Explanation:

ROS 2 coordinate convention (REP 103):

  • x: Forward (red axis)
  • y: Left (green axis)
  • z: Up (blue axis)

Interpreting xyz="0 0.2 0.5":

  • x = 0: No forward/backward offset
  • y = 0.2: 0.2 meters to the left (positive y)
  • z = 0.5: 0.5 meters up (positive z)

Visualization:

       torso
         |
         |--- (0, 0.2, 0.5)  shoulder joint location
         |
    upper_arm (child frame starts here)

For a humanoid:

  • Left shoulder: xyz="0 0.2 0.5" (positive y = left side)
  • Right shoulder: xyz="0 -0.2 0.5" (negative y = right side)

Important: <origin> is always relative to parent link frame, not world coordinates.

Reference: See urdf-basics.mdx:391-415 for coordinate transforms and REP 103.

Question 5: Joint Axis Direction​

You want a humanoid's knee to bend (like a human knee). The thigh is vertical (along z-axis). What should <axis xyz> be?

A) xyz="1 0 0" (x-axis, roll) B) xyz="0 1 0" (y-axis, pitch) C) xyz="0 0 1" (z-axis, yaw) D) xyz="0 -1 0" (negative y-axis)

Show Answer

Correct Answer: B) xyz="0 1 0" (y-axis, pitch)

Explanation:

Joint axis determines rotation direction:

AxisVectorRotation TypeHumanoid Example
x-axis1 0 0Roll (spin)Wrist twist, hip roll
y-axis0 1 0Pitch (up/down)Elbow, knee, ankle
z-axis0 0 1Yaw (left/right)Neck turning, waist

Knee joint example:

<joint name="knee" type="revolute">
  <parent link="thigh"/>
  <child link="shin"/>
  <origin xyz="0 0 -0.4" rpy="0 0 0"/>  <!-- bottom of thigh -->
  <axis xyz="0 1 0"/>  <!-- pitch rotation (y-axis) -->
  <limit lower="-2.356" upper="0" effort="80" velocity="2.0"/>
  <!-- -135 to 0 (knee bends backward only) -->
</joint>

Visualization (side view):

        z (up)
        |
   thigh
        
  ����������� y (left, axis of rotation)
       /
      / shin

Why y-axis? When you bend your knee, your shin rotates around a left-right axis (y in ROS), creating pitch motion (up/down).

Section 3: Humanoid Modeling​

Question 6: Mirrored Limbs​

You've defined a left arm starting at xyz="0 0.2 0.5" from the torso. What should the right arm's origin be?

A) xyz="0 0.2 0.5" (same as left) B) xyz="0 -0.2 0.5" (mirror y-coordinate) C) xyz="0 0.2 -0.5" (mirror z-coordinate) D) xyz="0 0.4 0.5" (double the y-coordinate)

Show Answer

Correct Answer: B) xyz="0 -0.2 0.5" (mirror y-coordinate)

Explanation:

Mirroring limbs for bilateral symmetry:

  • x (forward/backward): Usually same for left/right pairs
  • y (left/right): Opposite sign for mirroring
  • z (up/down): Usually same for left/right pairs

Example - shoulders:

<!-- Left shoulder -->
<joint name="left_shoulder" type="revolute">
  <parent link="torso"/>
  <child link="left_upper_arm"/>
  <origin xyz="0 0.2 0.5" rpy="0 0 0"/>  <!-- +y = left side -->
  <axis xyz="0 1 0"/>
  ...
</joint>

<!-- Right shoulder (mirrored) -->
<joint name="right_shoulder" type="revolute">
  <parent link="torso"/>
  <child link="right_upper_arm"/>
  <origin xyz="0 -0.2 0.5" rpy="0 0 0"/>  <!-- -y = right side -->
  <axis xyz="0 1 0"/>  <!-- same axis! -->
  ...
</joint>

Pattern for all paired limbs:

  • Shoulders: Mirror y-coordinate
  • Hips: Mirror y-coordinate
  • Arms/legs: Same length, mirrored attachment points

Common mistake: Mirroring the axis (xyz="0 -1 0"). The axis direction usually stays the same, only the position changes.

Reference: See urdf-basics.mdx:561-585 for mirrored limb patterns.

Section 4: Troubleshooting​

Question 7: Debugging URDF Errors​

You run check_urdf my_robot.urdf and get: "Joint 'elbow' parent link 'upper_arm' does not exist."

What's the most likely cause?

A) The elbow joint is defined before the upper_arm link B) The upper_arm link has incorrect inertia values C) The elbow joint should use type continuous not revolute D) The collision geometry is missing from upper_arm

Show Answer

Correct Answer: A) The elbow joint is defined before the upper_arm link

Explanation:

URDF parsing order matters: Links must be defined before joints that reference them.

WRONG (causes error):

<!-- Joint defined first -->
<joint name="elbow" type="revolute">
  <parent link="upper_arm"/>  <!-- upper_arm doesn't exist yet! -->
  <child link="forearm"/>
</joint>

<!-- Link defined after joint -->
<link name="upper_arm">...</link>
<link name="forearm">...</link>

CORRECT:

<!-- Links defined first -->
<link name="upper_arm">...</link>
<link name="forearm">...</link>

<!-- Joint references existing links -->
<joint name="elbow" type="revolute">
  <parent link="upper_arm"/>  <!-- upper_arm exists  -->
  <child link="forearm"/>     <!-- forearm exists  -->
</joint>

Best practice order:

  1. Define all materials
  2. Define base_link (root)
  3. Define child link
  4. Define joint connecting parent to child
  5. Repeat steps 3-4 for entire tree

Debugging workflow:

# Step 1: Validate XML and check for this error
check_urdf my_robot.urdf

# Output will show:
# Error: link 'upper_arm' is not defined
# at line: 47 in my_robot.urdf

Reference: See urdf-basics.mdx:621-650 for common URDF errors.

Section 5: Practical Application​

Question 8: Digital Twin Purpose​

Which of the following is NOT a primary use case for a robot digital twin (URDF model)?

A) Testing control algorithms without physical hardware B) Training machine learning models in simulation C) Compiling ROS 2 C++ code faster D) Visualizing robot state during debugging

Show Answer

Correct Answer: C) Compiling ROS 2 C++ code faster

Explanation:

Valid digital twin use cases:

  1. Algorithm development before hardware arrives (Option A)

    • Test motion planning, inverse kinematics, control loops
    • Example: MoveIt2 motion planning using URDF + simulation

  2. Machine learning training (Option B)

    • Generate synthetic training data
    • Example: Train vision models for object grasping in Isaac Sim

  3. Debugging and visualization (Option D)

    • See robot state in real-time via rviz2
    • Understand TF transforms and coordinate frames

  4. Safety testing

    • Simulate dangerous scenarios (falls, collisions)
    • Test emergency stop behavior

  5. Collaboration and documentation

    • Share robot specifications with team members
    • Auto-generate documentation from URDF

NOT a use case: Compilation speed (Option C)

  • URDF is a robot description, not code
  • Compilation speed depends on compiler settings, hardware, and code complexity
  • URDF has zero effect on C++ build times

Bonus: Complete System Design​

You're building a humanoid robot with:

  • 15 links (torso, head, arms, legs)
  • 14 revolute joints
  • 2 cameras (head-mounted, fixed)
  • 1 IMU sensor (torso-mounted, fixed)

What ROS 2 / URDF concepts would you use? (Select all that apply)

A) URDF to model kinematic chain B) Fixed joints for camera and IMU mounts C) TF tree to track all coordinate frames D) robot_state_publisher to broadcast transforms E) Services to update joint positions

Show Answer

Correct Answers: A, B, C, D (not E - services aren't the right pattern for joint updates)

Explanation:

Complete system architecture:

# 1. URDF model (A)
robot_description = """
<robot name="humanoid">
  <!-- 15 links with visual, collision, inertial -->
  <link name="base_link">...</link>
  <link name="torso">...</link>
  ...

  <!-- 14 revolute joints -->
  <joint name="waist" type="revolute">...</joint>
  <joint name="neck" type="revolute">...</joint>
  ...

  <!-- Fixed sensors (B) -->
  <link name="camera1">...</link>
  <joint name="camera1_mount" type="fixed">
    <parent link="head"/>
    <child link="camera1"/>
  </joint>

  <link name="imu">...</link>
  <joint name="imu_mount" type="fixed">
    <parent link="torso"/>
    <child link="imu"/>
  </joint>
</robot>
"""

# 2. robot_state_publisher (D) - publishes TF tree (C)
Node(
    package='robot_state_publisher',
    executable='robot_state_publisher',
    parameters=[{'robot_description': robot_description}],
    # Subscribes to /joint_states
    # Publishes /tf and /tf_static
)

# 3. Joint control (NOT services - use topics/actions)
# For position control, publish to /joint_states topic
# For trajectory control, use control_msgs/FollowJointTrajectory action

TF tree (C) generated:

base_link

      �� camera1 (fixed)

   �� imu (fixed)
   �� left_upper_arm

Why not E (services)?

  • Joint positions change continuously (30-100 Hz)
  • Services are for infrequent request-response patterns
  • Use topics (/joint_states) or actions (trajectory execution) instead

Best practice for joint control:

# Publisher for joint states
joint_state_pub = self.create_publisher(JointState, '/joint_states', 10)

# Or use ros2_control for hardware interface
